## 4. 需要优先处理的问题

### P0-1. `check_https_direct()` fallback 结果字段与 `enrich()` 不兼容

位置：`build_dataset.py`

当前 fallback 成功时返回类似：

```python
{
  "success": True,
  "supports_ipv4": True,
  "country": "US",
  "colo": None,
  "cf_bot_score": 95,
  "method": "direct_https"
}
```

但 `enrich()` 主要读取：

```python
item["probe_results"]["ipv4"]["exit"]["botManagement"]
```

结果：fallback 成功的 IP 很可能无法正确获得 `country / asn / bot score`，后续 `TARGET_COUNTRIES=US` 过滤时可能被排掉，或者排序质量异常。

建议：

- 要么把 `check_https_direct()` 的返回结构改成模拟 `cmliu` 的 `probe_results.ipv4.exit` 结构；
- 要么让 `enrich()` 同时兼容 fallback 扁平字段。

优先级：**最高**。这是当前最实质的代码逻辑问题。

### P1-1. Worker secret 命名在线上没问题，但仓库层面仍混乱

现状：

- Worker 代码读取：`PROXYIP_SECRET`
- Cloudflare 线上 secret：`PROXYIP_SECRET` 已存在
- GitHub Actions / README / Python 脚本使用：`PROXYIP_HMAC_SECRET`

所以线上不是故障，但对后续维护者很容易误解。

建议：

方案 A：统一全部叫 `PROXYIP_HMAC_SECRET`。

方案 B：保留现状，但 README 明确写：

```text
GitHub repo secret: PROXYIP_HMAC_SECRET
Cloudflare Worker secret: PROXYIP_SECRET
两者值必须相同。
```

优先级：**高**。

---

### P1-2. `/robots.txt` 实际内容与 Worker 代码/README 不一致

Worker 代码写的是：

```text
User-agent: *
Disallow: /
```

但线上实际 `/robots.txt` 返回 Cloudflare Content Signals 长文本，不是 Worker 代码中的内容。

可能原因：Cloudflare 的 managed robots/content signals 功能优先于 Worker 响应。

影响：

- README 说“robots.txt 禁止抓取”，但线上实际不一定是这个明确内容；
- 如果你依赖 `/robots.txt` 做反爬声明，这里需要确认 Cloudflare 设置。

建议：

- 如果要强制 `Disallow: /`，去 Cloudflare 检查是否启用了 managed robots / AI crawler content signals；
- 或接受 Cloudflare 内容信号，但 README 改成“由 Cloudflare Content Signals + X-Robots-Tag 共同控制”。

优先级：**高**。

---

### P1-3. `/health` 和 `/stats` 未鉴权，信息暴露偏多

现在无 token 可访问：

- 当前主 IP
- standby_count
- 总数量
- checked_at
- ASN 分布
- Colo 分布
- 延迟分布

这不算严重秘密，但对“降低被爬取和滥用”目标来说暴露偏多。

建议：

- `/health` 公开版只返回：`ok / checked_at / count / stale`；
- 新增 `/health/full` 或 `/stats` 需要 cookie/token。

优先级：**中高**。

---

### P1-4. 缺少数据新鲜度熔断

当前 `/health` 只要 KV 有数据就 `ok: true`，没有判断 `checked_at` 是否过期。

建议：

- 超过 6 小时：`stale: true`；
- 超过 12 小时：`ok: false`；
- 首页显示明显警告；
- GitHub Actions 连续失败时不要让旧数据看起来仍然健康。

优先级：**中高**。

---

### P1-5. 当前主 IP 只要还能用就一直保留，缺少质量下限

当前策略是“稳定优先”：只要 previous IP 仍在 valid 列表，就继续保留。

问题：

- 如果当前 IP 仍可用，但延迟暴涨、bot score 下降、ASN 风险变高，仍可能继续保留；
- 对 AI / CDN 稳定访问来说，“可用”不等于“好用”。

建议增加最低质量线，例如：

- `cf_bot_score >= 80`
- `corporate_proxy == false`
- `verified_bot == false`
- `latency_ms <= 某阈值`
- `country in TARGET_COUNTRIES`

低于质量线时进入 failover 候选，但可加冷却时间避免频繁切换。

优先级：**中高**。

## 5. 中优先级优化

### P2-1. `rateBuckets` 无清理逻辑

上一份报告说“长期必崩”有点夸张，因为 Worker isolate 会被 Cloudflare 回收，不是传统常驻进程。

但问题确实存在：同一个 isolate 生命周期内，`rateBuckets` 只增不删。

建议：

- 当 `rateBuckets.size > 5000` 时清理过期 bucket；
- 或每 N 次请求清理一次；
- 或加最大容量。

---

### P2-2. 304 响应没有统一安全 header

当前命中 ETag 时直接返回：

```javascript
new Response(null, { status: 304 })
```

缺少：

- `cache-control`
- `x-robots-tag`
- `x-content-type-options`
- `referrer-policy`
- `x-frame-options`

建议：304 也走 `withHeaders()`，并带上 `etag`。

---

### P2-3. `HEAD` 请求返回 405，对监控工具不友好

`curl -I /current.txt` 和 `curl -I /robots.txt` 返回 405，因为 Worker 只允许 GET。

这不是 bug，但很多监控工具默认用 HEAD。

建议：

- 支持 `HEAD`，返回与 GET 相同 headers、空 body；
- 或 README 标注健康检查必须用 GET。

---

### P2-4. `/v2ray.txt` 命名误导

当前只是：

```javascript
btoa(ips.join("\n"))
```

也就是“base64 编码的 IP 列表”，不是标准 V2Ray 订阅。

建议：

- 如果保留现状，改名 `/base64.txt`；
- 如果要叫 `/v2ray.txt`，就生成真正的 V2Ray/VLESS 节点格式。

---

### P2-5. KV 上传了多个 Worker 没用到的 key

Worker 实际只读取：

```text
result_json
```

但 KV 里还有：

```text
current_json
current_txt
history_json
standby_txt
state_json
```

这不是错误，但会增加同步步骤和失败面。

建议：

- 如果这些 key 是备用/调试用途，在 README 写明；
- 否则只保留 `result_json`。

---

### P2-6. `full.json` 太大，仓库会越来越重

`docs/full.json` 包含 `all_results`，信息量大；KV 上传时已经 strip，但 GitHub 仓库仍提交完整文件。

建议：

- commit 精简版 `full.json`；
- 全量调试结果作为 GitHub Actions artifact 保存；
- 或拆成 `full.json` + `debug-results.json`，后者不提交。

---

### P2-7. 域名和 zone 硬编码

多个文件写死：

- `leilaomi.cc.cd`
- `proxyip.leilaomi.cc.cd`
- `list.leilaomi.cc.cd`

建议支持环境变量覆盖：

```text
PROXYIP_ZONE_NAME
PROXYIP_RECORD_NAME
PROXYIP_LIST_DOMAIN
```

这样同名项目复制到其他域名时更稳。

## 6. 低优先级建议

### P3-1. GitHub Actions 可加 npm/wrangler 缓存

目前每次安装 `wrangler@4`。

不影响正确性，只影响速度。

---

### P3-2. 加 CSP / Permissions-Policy

当前已有：

- HSTS
- X-Frame-Options
- X-Content-Type-Options
- Referrer-Policy
- X-Robots-Tag

建议补充：

```text
Content-Security-Policy
Permissions-Policy
```

---

### P3-3. ETag 建议改为内容 hash

当前 ETag 使用 `checked_at`。

问题：数据没变但重新生成时间变了，缓存也失效。

建议：用 `current_ip + valid_ips` 的 hash 生成 ETag。

---

### P3-4. `stable_score` 和 `rank_key` 重复

当前两套排序/评分逻辑并存，权重略有差异。

建议长期统一成一个评分模型，避免解释成本。

## 8. 推荐修复路线

### 第一轮：小改动、高收益

1. 修 `check_https_direct()` 与 `enrich()` 字段兼容。
2. README 改掉过期动态数字。
3. README 明确 `PROXYIP_HMAC_SECRET` 与 `PROXYIP_SECRET` 的关系。
4. `/health` 加 stale 判断。
5. `rateBuckets` 加清理逻辑。
6. 304 响应统一安全 headers。

### 第二轮：安全与可维护性

1. `/stats` 改为鉴权访问。
2. `/health` 分公开版和详细版。
3. 支持 HEAD 请求。
4. 域名/zone/record 支持环境变量覆盖。
5. 确认 Cloudflare robots/content signals 设置，修 README 或关掉覆盖。

### 第三轮：产品化

1. 真实 V2Ray/VLESS 订阅格式。
2. GitHub Actions artifact 保存完整 debug 数据。
3. 失败通知：连续失败 2 次后通知。
4. 当前主 IP 质量下限 + failover 冷却时间。

## 9. 最终判断

项目当前线上状态是健康的：Worker、KV、DNS、Custom Domain、secret 都配好了，`proxyip.leilaomi.cc.cd` 当前正确指向 `43.170.25.96`，`list.leilaomi.cc.cd` 也正常从 KV 分发数据。

真正需要处理的不是“线上已经坏了”，而是：

1. fallback 检测逻辑有结构性缺陷；
2. 文档和实际状态不同步；
3. 健康检查还没有“过期数据熔断”；
4. 公开统计信息偏多；
5. 后续复制/维护时 secret 和域名硬编码容易踩坑。

建议后续先做第一轮修复，不要一次性大改。
